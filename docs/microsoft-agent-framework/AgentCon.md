# 从一份简历到未来软件形态

## 一、从一份简历说起

### 用"共识"开场

你可以这样开始：

> 今天我们聊 Agent、聊架构、聊未来软件，
> 但我想先从一个所有人都见过的东西说起——
> 一份简历。

然后你可以停顿一下：

> 不管你是工程师、HR、创业者
> 
> 每个人都"理解过简历"
> 
> 有时候甚至一眼就能判断一个人是否合适

### 人类 vs 计算机的"理解差异"

接着你转一个角度：

> 对人类来说，理解一份简历几乎是"本能行为"。
> 
你可以举一些非常生活化的例子：

- "2018–2023，后端工程师"
- "负责核心系统设计"
- "熟悉 .NET / Java / 分布式系统"

人类会自动完成很多推理：

- 工作年限
- 技术深度
- 角色定位
- 成长轨迹

然后你反问一句（非常重要）：

> 那如果这份简历交给计算机呢？

### 把"简历"拆成计算机眼里的样子

你可以开始技术视角切入：

> 从计算机的角度看，简历其实是一种 **高度不友好的输入数据**。

然后顺着这些点展开（不是念 bullet）：

**非结构化**
- PDF、Word、图片
- 表格、段落、列表混在一起

**多语言**
- 中文、日文、英文混用
- 同一个职位，不同国家写法完全不同

**强语义依赖**
- "负责""参与""主导"差异极大
- 技术栈的深度往往是隐含的

你可以给一句工程师会点头的总结：

> 对系统来说，它拿到的不是"一个人"，
> 而是一堆 **格式混乱、语义模糊、上下文缺失的内容**。


> 所以我们慢慢意识到一件事：
> 简历解析并不是一个文本问题。
> 
> 而是：
> - 一个理解问题
> - 一个推理问题
> - 一个跨语言、跨语境的语义对齐问题

这一句是承上启下的关键：

> 这也意味着——
> 任何一个试图"用一个 Prompt 解决简历解析"的方案，
> 从一开始就站错了抽象层级。

然后你自然过渡到第二部分：

> 那在 Linkora 里，我们到底是怎么做这件事的？
> 为了"理解一份简历"，我们实际上动用了多少 AI？


## 二、在 Linkora 中如何完成一份简历解析？

在讨论 AI 架构之前，我们先澄清一个常见误解。复杂的问题，并不一定意味着需要大量 AI 能力。

### 一个反直觉但真实的结论


### 利用AI 文档解析（PDF / Word）

**解决的问题：** 让系统"看见"简历的内容。

- 输入来源不确定：PDF、Word、不同模板
- 文本、段落、列表混合存在
- 需要尽可能还原原始语义结构

这一阶段的目标并不是理解简历，而是将不可处理的文件，转化为稳定、可处理的文本输入。

### 文本结构化为 JSON

**解决的问题：** 让系统"处理"简历内容。

将非结构化文本映射到明确的数据模型。

输出必须满足：
- 字段稳定
- 可反序列化
- 可存储、可追踪

这是从"文本"到"数据"的关键一步，也是系统可扩展性的基础。

### 向量化（Embedding）

**解决的问题：** 让系统"记住"这个人。

将结构化内容转为语义向量，用于后续的：
- 搜索
- 匹配
- 推荐

向量化并不负责理解业务，而是为系统提供一种可计算的语义表示。

### Agent（决策与控制层）
### Agent（决策与控制层）

#### 执行与协作层（Action / Execution Layer）
→ MCP / Function Call / A2A（执行层）

Agent 层位于模型能力与系统执行之间，负责将模型提供的智能能力，转化为可控、可持续的系统决策。

这一层关注的不是"模型怎么调用"，而是**系统在任意时刻应该做什么、是否继续、如何推进**。

#### Agent 在系统中的核心职责

在 Microsoft Agent Framework 中，Agent 不是一次性推理结果，而是一个持续参与系统运行的决策单元，主要承担以下职责：

- 基于上下文使用模型进行推理与判断
- 决定下一步是否执行、执行什么能力
- 协调多个能力或多个 Agent 的协作
- 对系统状态与执行进度负责

> Agent 解决的是"决策与控制"问题，而不是"智能强度"问题。

#### 上下文与多级记忆（Context / Thread / Persistent Agent）

MAF 将"记忆"从 Prompt 中抽离，提升为系统级能力。

- **Context** — 单次决策所需的即时上下文
- **Thread** — 一个任务或会话的连续执行状态
- **Persistent Agent** — 跨请求、跨时间存在的 Agent 状态

这使得 Agent 不再依赖一次请求完成所有事情，而是能够在复杂、长期任务中持续做决策。

#### Agent 生命周期与编排能力

MAF 将 Agent 视为一等系统对象，并提供完整的生命周期管理能力：

- Agent 可以被创建、挂起、恢复
- 决策过程不再绑定某一次调用或进程
- 多 Agent 可被统一编排与调度

这使得系统从"线性流程"，演进为以决策为中心的执行模型。

#### Agent 与执行层的职责边界

在 MAF 架构中，Agent 不直接产生副作用。

**Agent 负责：**
- 判断是否需要行动
- 选择合适的能力或协作对象

**具体执行：**
- 由 Function Call、MCP、A2A 等执行层完成

> Agent 决定"要不要做"，执行层负责"具体怎么做"。

这一边界确保了系统的可控性与可治理性。
